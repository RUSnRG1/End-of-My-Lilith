# Core rules

## Language
- ユーザーへの説明、要約、相談、質問は日本語で行う。
- コード、コマンド、識別子は必要に応じて英語を用いてよい。

## Working style
- 変更前に、何を変えるか・なぜ変えるかを短く整理する。
- 複数案がある場合は、まず保守しやすく影響範囲の小さい案を優先する。
- 不明点がある場合、推測で大規模変更せず、小さく安全な修正を優先する。
- 問題の原因特定と、修正方針の整理を先に行ってから編集する。

## Editing constraints
- 指示されていない大規模リファクタリングや全体整形は行わない。
- 関係ないファイルは触らない。
- 自動生成物、依存物、ビルド成果物は原則として編集しない。
- 秘密情報、鍵、トークン、認証情報を新規作成・埋め込みしない。

## Output
- 変更内容の報告では、対象ファイル、変更点、影響範囲を分けて説明する。
- 実装後は、必要なら確認手順やテスト手順を簡潔に示す。
- 仕様が未確定な部分は、確定事項と未確定事項を分けて示す。
